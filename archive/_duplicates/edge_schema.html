M6.2 Relationship / Edge Schema — Directionality & Semantics

Micro-steps

  1. Create `schemas/edge.schema.json` and `schemas/edge_type_registry.json`.
  2. Define edge fields: id, type, from_id, to_id, directed, weight, confidence, provenance, evidence_refs.
  3. Define edge typing rules (allowed from/to entity types per edge type).
  4. Implement `make_edge_id()` (stable hash of from/to/type + evidence key).
  5. Implement `validate_edge_types()` with clear error messages.
  6. Sort edges deterministically: (from_id, type, to_id, id).
  7. Emit `edges.json` and `edge_stats.json`.

Edge cases

  1. Self-referential edges (allowed only for explicit types).
  2. Multiple edges between same nodes with distinct evidence.
  3. Cycles (allowed; must be detectable and reportable).

Definition of Done

  1. Edge registry enforces typing constraints.
  2. Edge IDs are stable; duplicates are detectable and reported.
  3. Edges output is deterministically ordered.
