M6.1 Entity Schema — IDs, Types, Fields

Key decisions to lock

  1. Entity types list and required fields per type.
  2. Stable ID strategy (uuidv5 / hash-of-normalized-key) and migration strategy.
  3. Display name vs canonical name vs slug rules.

Micro-steps

  1. Create `schemas/entity.schema.json` and `schemas/entity_type_registry.json`.
  2. Define required core fields: id, type, name, slug, created_at, updated_at, provenance.
  3. Define optional fields: description, tags, attributes, canonical_refs, timeline_refs.
  4. Implement `normalize_entity_key()` (casefold, whitespace, punctuation rules).
  5. Implement `make_entity_id()` (stable, deterministic; no random UUIDs).
  6. Write loader `load_entities()` that validates schema and sorts deterministically.
  7. Emit `entities.json` with stable field ordering (canonical serialization).

Edge cases

  1. Multiple spellings, accents, punctuation variants.
  2. Same name across different types (Person vs Place).
  3. Entities introduced before fully known (placeholder stubs).

Definition of Done

  1. Schema validation fails fast with actionable errors.
  2. IDs are stable across rebuilds and independent of file order.
  3. Entity JSON output is deterministically ordered and formatted.
