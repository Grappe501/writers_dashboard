M0.5 Test Strategy — Matrix, Fixtures, Golden Outputs

Goal

Tests must protect architecture: day-based timeline, spoiler gating, determinism, and schema validity.

Fixture Set

  • ToyBook: 5–10 items, 3 entities, 1 conflict, 1 override
  • Book1Window: Dec 1 → Jan 3 real-time anchor window; enough data to exercise rules
  • EdgeCaseSet: empty day, duplicate entities, time collisions, alias loops

Golden Outputs

  • Graph export (entities/edges) with stable IDs
  • Search index artifacts
  • Standalone HTML reader output
  • Build manifest + reports

Build Order (micro-steps)

  1. Define fixtures as immutable input files.
  2. Generate golden outputs once and store under /golden.
  3. Add test harness to rebuild and diff against golden outputs.
  4. Add per-module unit tests for parsers/validators.
  5. Add integration tests across M4→M6→M7→M8 pipeline.

Definition of Done (DoD)

  • Fixtures exist and are immutable.
  • Golden outputs exist and are diffed in CI.
  • Integration tests cover the full build pipeline on fixtures.
