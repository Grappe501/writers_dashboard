M2.5 Override Layer — Bounded Bypasses + Audit

Goal

Allow author intent without destroying the safety rails.

Boundaries

  • Overrides must target specific rule ids (not ‘disable validation’).
  • Overrides must declare scope (items/entities/days).
  • Overrides must be time-bounded or release-bounded where possible.
  • Overrides are visible in exports and reader metadata.

Build Order (micro-steps)

  1. Implement override matcher: (rule_id + scope) → allow/deny.
  2. Annotate conflict outputs with override status.
  3. Emit override summary into build manifest.
  4. Add CI rule: critical conflicts cannot be overridden unless specifically allowed by policy.

Definition of Done (DoD)

  • Overrides only bypass declared rule ids.
  • Build manifest includes override summary and counts.
